Product Catalog and CSV Admin Upload Spec

Purpose

This document defines:
	1.	The initial product categories and starter product list for the website.
	2.	The content fields required for product pages.
	3.	A CSV-upload workflow through a hidden admin URL so new products can be added quickly.
	4.	A CSV-based update workflow so new images can also be added to existing products.

This is meant for the phase-1 UI-first catalog, where products can be shown immediately and operational workflows can remain manual until a full backend is plugged in.

5. Hidden Admin CSV Upload Workflow

A hidden hardcoded admin URL should be created for uploading product CSV files.

Suggested hidden admin URL

/admin/a7f3c91b6d4e8f20c5b9/product-csv-upload

This URL should not be linked from the public navigation.

Purpose of this page

This page allows the internal team to:
	1.	Upload a CSV file to add brand-new products.
	2.	Update existing products by matching product_id or sku.
	3.	Add or replace image URLs for existing products.
	4.	Bulk update pricing, tags, visibility, and product descriptions.

⸻

6. CSV Upload Modes

The admin upload page should support 3 modes.

Mode 1: Add New Products

If a product row contains a new product_id or new sku, the system should create a new product.

Expected result
	•	new product is added
	•	category and subcategory are mapped
	•	product page data becomes available
	•	hero image and gallery images are stored
	•	product appears on catalog pages based on visibility flags

Mode 2: Update Existing Products

If a row contains an existing product_id or existing sku, the system should update the product.

Updatable fields
	•	pricing
	•	descriptions
	•	tags
	•	visibility switches
	•	SEO fields
	•	CTA fields

Mode 3: Image-Only Updates

A CSV can also be uploaded just to add or replace images on existing products.

Example use
	•	A product already exists on the site.
	•	The team gets better image links later.
	•	The CSV is uploaded with matching product_id plus new image URLs.
	•	The system replaces only the media fields without disturbing other fields.

⸻

7. CSV Image Update Rules

Image matching logic

The system should first try:
	1.	product_id
	2.	then sku
	3.	then optionally exact slug

Image replacement rules
	•	if a new hero_image_url is provided, replace hero image
	•	if gallery URLs are provided, replace only the corresponding gallery slots
	•	if a field is blank, keep existing image
	•	if a special clear_gallery flag is true, remove all gallery images before applying new ones

Validation rules
	•	image URL must be a valid URL
	•	duplicate image URLs should be prevented within the same product
	•	if hero image is blank and product is new, reject the row
	•	if product is active and has no hero image, flag as incomplete

9. Product Completion Rules

A product should be considered ready for public display only if these minimum conditions are met:
	•	product_name exists
	•	category exists
	•	subcategory exists
	•	short_description exists
	•	hero_image_url exists
	•	at least 2 total images exist
	•	price_display exists or enquiry_mode is set to quote-based
	•	slug exists
	•	is_active is true

If a product is incomplete:
	•	it may still exist in admin
	•	but it should not appear publicly unless forced by admin override

14. Final Implementation Intent

This system should allow the business to:
	•	launch quickly with a broad product catalog
	•	avoid manual code changes for every new product
	•	add products through CSV
	•	enrich products later with better images and content
	•	update existing products through CSV
	•	keep the public website scalable without forcing a full backend from day one

The CSV-upload layer is the bridge between a UI-first MVP and a future full ecommerce/catalog backend.
